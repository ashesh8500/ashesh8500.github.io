The one-minute explanation

Fractal is a paper-trading research engine. It watches a basket of stocks, creates structured “beliefs” about what looks interesting, and separately runs a live trading process that can submit orders to Interactive Brokers — but only through risk-budget checks and audit logs.

1. Eyes

YFinance + IBKR feed the system with prices, account state, fills, and position information.

2. Brain

The cognition pipeline turns raw prices into artifacts, hypotheses, seven role perspectives, belief states, and attention routes.

3. Hands

The live trader connects to IB Gateway and submits/records orders from enabled strategies, currently paper mode.

The clean analogy

Like a driving instructor

  • The cognition layer says: “I see a car, a red light, and a risky merge.”
  • The risk governor says: “Even if you want to turn, are you allowed?”
  • The live trader is the steering wheel connected to the simulator.

Important boundary

  • A cognition “proposal” is not the same thing as an executed trade.
  • An approved risk decision is not enough unless an execution client places and fills the order.
  • SQLite tables are the audit trail: beliefs, intents, risk decisions, fills.

What each stage means

The live thesis: “I see AI doing well”

This is the currently encoded top-down hypothesis. In code it is the switchable thesis prompt AI is the future. The important part is not the phrase itself; it is how the system turns a broad human belief into a ranked, risk-governed portfolio target.

Figure 2. The BCE logic

Figure 2. BCE means Beneficiary / Compute Provider / Enabler. Each lens is scored independently first. Only afterward does overlap become a portfolio signal.

Walkthrough: one belief becomes a portfolio

1
Start with the human hypothesis.
The system receives the thesis prompt: AI is the future. It treats that as a switchable belief, not a permanent hardcoded strategy. Tomorrow the same process could run for “nuclear renaissance” or “India consumer compounding.”
2
Split the thesis into orthogonal lenses.
B = companies whose products capture AI value; C = companies that operate or finance compute; E = companies that enable the physical stack: GPUs, foundries, semi equipment, optics, power, cooling, packaging.
3
Score every symbol independently inside each lens.
The per-lens formula is 0.62 × thesis_quality + 0.14 × financing_quality + 0.12 × optionality + 0.12 × momentum − volatility_penalty. At this stage, overlap is deliberately ignored so the model does not double-count the same story too early.
4
Only then accumulate overlap.
The controller computes 0.58 × best_lens + 0.42 × average_lens + overlap_bonus + financing_bonus + optionality_bonus − caution_penalty. The overlap bonus is 0.00 for one lens, 0.08 for two lenses, and 0.18 for all three. This is why GOOGL / NVDA / AVGO can rank highly: they show up from multiple independent directions.
5
Convert scores into target weights.
The selected top candidates are sorted by final score. Weight is proportional to score, capped by max_weight; unused capped weight stays cash rather than forcing exposure.
6
Execution is separate and gated.
The thesis meta-controller compares targets to the current book. It sells decayed names, trims >1.3× target, adds <0.7× target, and enters new names only through structured trade_intents and risk_decisions. A thesis score is not itself an order.

What this generated logic is trying to avoid

  • Not just “buy NVDA because AI.” NVDA gets a strong score, but with a crowding/valuation caution.
  • Not one giant AI bucket. MSFT can beat more “AI-looking” names because financing quality matters; self-funded capex is part of the thesis.
  • Not overlap too early. B/C/E are scored separately first, then overlap is rewarded after the independent evidence exists.
  • Not automatic trading. The final portfolio target must still pass book comparison and risk governance before anything hits IBKR.

What worked
  • Auditability exists. Beliefs, attention, intents, risk decisions, account state, and fills are in SQLite stores.
  • Paper/live boundary is explicit. Config points to IB Gateway paper port 4002; switching live is an infrastructure change, not a hidden strategy flag.
  • Risk governor blocks existential mistakes. It checks symbol allowlist, stale data, loss budgets, open notional, leverage, shorts/options settings.
  • Cognition is now calmer. Latest sweep shows monitor postures rather than all-symbol advisor review.
Known flaws / weak spots
  • Cognition is mostly advisory telemetry. It does not yet have a complete closed-loop learning lifecycle: proposals, gates, outcomes, and advisor feedback are sparse or zero in the current stores.
  • Two brains can diverge. The cognition pipeline and the live trading/meta-controller are adjacent but not fully unified; an advisor belief is not automatically the source of a live order.
  • Gateway positions table is empty right now. Fills exist, and account state exists, but current position display can require alternate ledgers/IBKR snapshots.
  • YFinance is not execution-grade. Useful for cognition and scans, but live execution decisions need fresh IBKR data checks.
  • Existing positions may be legacy. Some fills came from autonomous allocator v2 / meta-controller history, not necessarily the current thesis or current cognition output.
Questions to ask when iterating
  1. Which component made this decision: cognition, thesis allocator, noise script, live momentum, or legacy allocator?
  2. Was there a structured intent, a risk decision, an order, and a fill — or only one of those?
  3. Did the trade have an exit thesis before entry?
  4. Did the system learn from the outcome, or just record it?
  5. Can Dad understand the reason in one sentence?
