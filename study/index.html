Critical path
a), b) · 10.0pts
▼

Consider the following SystemVerilog code:

always_comb begin
    x1_next = func1(x1); // delay = 2.3 ns

    // mux delay = 0.4 ns
    if (x1 > 2) begin  // delay = 0.5 ns
        x2_next = func2(x1);  // delay = 3.7 ns, depends on x1
    end else begin
        // delay = 2.1 ns, depends on x1_next, not x1
        x2_next = func3(x1_next);  
    end
end
always_ff @(posedge clk) begin
    x1 <= x1_next;
    x2 <= x2_next;
end

The if condition is implemented with a multiplexer with the condition and two branches evaluated in parallel. The multiplexer adds an additional delay of 0.4 ns.

  1. Determine the critical path delay for the combinational logic that produces x1_next and x2_next.
  2. What is the maximum clock frequency for this design assuming clock-to-Q delay is 0.2ns and setup time is 0.1ns?
📐 Visual walkthrough
  1. The critical path delay is the longest delay through the combinational logic. After each rising edge, we compute the maximum propagation delay for each signal:
    • x1_next: the delay is simply the delay of func1, which is 2.3 ns.
    Now for x2_next, since it is produced by an if statement, it will be realized in hardware via a multiplexer. The select condition and the two branches are all computed in parallel.
    • The select condition is x1 > 2, which has a delay of 0.5 ns from the beginning of the clock cycle (since it depends on x1).
    • The first branch through func2 has a delay of 3.7 ns from the beginning of the clock cycle (since it also depends on x1).
    • The second branch through func3 has a delay of 2.1 ns after x1_{next} is stable, so its delay is 2.3 + 2.1= 4.4 ns.
    • Hence, the last of these three signals to become stable is the second branch through func3, which becomes stable at 4.4 ns. The multiplexer adds an additional delay of 0.4 ns, so the total delay for x2_next is 4.8 ns.
    We see the critical path is: 
    x1 --> func1 --> x1_next --func3--> mux -> x2_next
    with a total delay of 4.8 ns.
  2. The clock period must satisfy \[ \begin{aligned} T_{clk} \geq T_{cp} + T_{clk-Q} + T_{setup} \\ & \geq 4.8 \text{ ns} + 0.2 \text{ ns} + 0.1 = 5.1 \text{ ns} \end{aligned} \] The maximum clock frequency is the reciprocal of the clock period, approximately 196.1 MHz (1 / 5.1 ns).
💡 Key Insight: Mux branches are parallel — take the max, not the sum. The critical path is x1 → func1(2.3) → func3(2.1) → mux(0.4) = 4.8ns. Max clock = 1/(4.8+0.2+0.1) = 196.1 MHz.
📋 Grading Notes: - The most common mistake is to ignore the parallelism in the mux implementation and add the delays of func2 and func3 as if both were on the same path. They are alternative data inputs to the mux, so the mux output depends on the maximum arrival time, not the sum of branch delays. - The critical path is through x1 to func1 to x1_next to func3 to the mux to x2_next, for a total delay of 4.8 ns. - Student may describe the critical path without explicitly naming every signal, as long as it is clear that they are considering the path through func1 and func3 rather than func2, and they include the mux delay. - If the student identifies the correct longest branch delay of 4.4 ns but forgets the 0.4 ns mux delay, award partial credit for the path analysis but not full credit for part (a). - Some students may identify the critical path directly without explicitly analyzing every shorter path. That is acceptable as long as the path and total delay are correct. - For part (b), accept equivalent frequency expressions and reasonable rounding. Units may be omitted if the intent is clear. - No explanation is required if the student gives the correct critical path delay and maximum clock frequency.

Consider the following SystemVerilog combinational code with two inputs, \(x_1\) and \(x_2\):

if (x1 > 2) begin  // Delay to evaluate condition = 0.5 ns
    z1 = func1(x1);  // propagation delay of 4 ns
end else begin
    z1 = func2(x1);  // propagation delay of 2 ns
end
z2 = func3(x2);   // propagation delay of 3 ns
y = func4(z1, z2);  // propagation delay of 1.5 ns

The if condition is implemented with a multiplexer, meaning both functions \(f_1\) and \(f_2\) are evaluated in parallel, but only one of their outputs is selected based on the value of \(x_1\). The multiplexer has a propagation delay of 0.8 ns. Suppose that before time 0, all signals are stable, and then at time 0, both \(x_1\) and \(x_2\) change value simultaneously. At what point do the following signals become stable (i.e., no longer unknown):

  1. \(z_1\)
  2. \(z_2\)
  3. \(y\)
📐 Visual walkthrough
  1. The signal \(z_1\) will become stable at time 4.8 ns. Since func1 and func2 are evaluated in parallel and the mux must wait until the selected input and select signal are valid, the worst-case stable time is the slower function delay of 4 ns plus the mux delay of 0.8 ns.
  2. The signal \(z_2\) will become stable at time 3 ns, which is the propagation delay of func3.
  3. The signal \(y\) will become stable at time 6.3 ns. The signal y depends on \(z_1\) and \(z_2\). Since \(z_1\) becomes stable at time 4.8 ns and \(z_2\) becomes stable at time 3 ns, func4 can produce a stable output 1.5 ns later, at time 6.3 ns.
💡 Key Insight: A mux with N inputs evaluates all N paths in parallel. The total delay = max(path delays) + mux delay.
📋 Grading Notes: - For part (a), full credit requires the worst-case propagation delay for z1, which is 4.8 ns. Do not accept an input-dependent answer such as 2.5 ns or 4.8 ns depending on the branch as fully correct. - For part (a), if the student correctly reasons that the slower branch determines the propagation delay but forgets the 0.8 ns mux delay and gives 4 ns, award partial credit. - For part (c), y depends on z1 and z2, not on x1. A common error is to wait for z1 and then add the func4 delay, which leads to the incorrect answer 5.5 ns. - Units may be omitted with no loss of points. - No explanation is required as long as the correct stable times are given. If the student provides an explanation, it can be used to award partial credit if the final answer is not fully correct.

Two cycle datapath
a), b), c) · 11.0pts
▼

Consider the following SystemVerilog code with an input x:

always_comb begin
    xsq = xreg*xreg;  
    axsq = a*xsq;  
    bx = b*xreg;  
    if (xreg > 10) begin 
        y = axsq + bx;  
    end else begin
        y = bx;  
    end
end
always_ff @(posedge clk) begin
    xreg <= x;
end

Assume the following delays:

  • Multiplication: 2 ns
  • Addition: 1 ns
  • Comparison xreg > 10: 0.5 ns
  • Register update: 1 ns
  • Multiplexer (for if statement): 0.4 ns

Answer the following:

  1. The current code computes y in a single cylce. What is the propagation time after the rising edge of the clock for the signal y to become stable (i.e., no longer unknown)?
  2. Rewrite the code registering xsq, bx, and gt10=xreg>10 to create a two cycle datapath for computing y.
  3. What is the critical path delay for the two cycle datapath?
📐 Visual walkthrough
  1. In the original single-cycle datapath, xreg becomes stable 1 ns after the rising edge because of the register update delay. From there:
    • xsq = xreg*xreg is stable at 1 + 2 = 3 ns,
    • axsq = a*xsq is stable at 3 + 2 = 5 ns,
    • bx = b*xreg is stable at 1 + 2 = 3 ns,
    • the condition xreg > 10 is stable at 1 + 0.5 = 1.5 ns.
    The slower data branch into the mux is the branch axsq + bx, which becomes stable at 5 + 1 = 6 ns. The select signal is ready earlier, at 1.5 ns, and the other data input bx is ready at 3 ns. Therefore the mux output y becomes stable after the latest arriving mux input plus the mux delay: \[ t_y = \max(6, 3, 1.5) + 0.4 = 6.4\ \text{ns} \] So y becomes stable 6.4 ns after the rising edge.
  2. One correct two-cycle implementation is to compute and register xsq and bx in the first cycle, and also register the branch decision so that the control stays aligned with those data values: 
    logic xsq, bx, gt10;
logic xsq_reg, bx_reg, gt10_reg;
logic axsq;

always_comb begin
    xsq = xreg * xreg;
    bx = b * xreg;
    gt10 = (xreg > 10);

    axsq = a * xsq_reg;
    if (gt10_reg) begin
        y = axsq + bx_reg;
    end else begin
        y = bx_reg;
    end
end

always_ff @(posedge clk) begin
    xreg <= x;
    xsq_reg <= xsq;
    bx_reg <= bx;
    gt10_reg <= gt10;
end
    Any functionally equivalent answer that delays the control path so it matches the registered data path is acceptable.
  3. For the two-cycle datapath, the two stages have delays:
    • Stage 1: from xreg to xsq_reg or bx_reg, the longest path is 1 + 2 = 3 ns.
    • Stage 2: after the next rising edge, xsq_reg, bx_reg, and gt10_reg are available after 1 ns. Then axsq = a*xsq_reg takes 2 ns more, the addition takes 1 ns more, and the mux adds 0.4 ns. So the longest stage-2 delay is \[ 1 + 2 + 1 + 0.4 = 4.4\ \text{ns} \]
    Therefore the critical path delay for the two-cycle datapath is 4.4 ns.
💡 Key Insight: Splitting logic across two cycles allows higher clock frequency. Each cycle must complete within tclk − tsetup − tclk-Q.

Loop carry dependency
a), b) · 10.0pts
▼

Consider the following two loops. Each loop contains a loop-carried dependency that affects the minimum possible initiation interval (II) when pipelined in Vitis HLS.

Loop A

sum = 0;
ind = 0;
for (int i = 0; i < 100; i++) {
    sum += x[ind];
    ind = ind_next[ind];
}

Loop B

sum = 0;
ind = 0;
for (int i = 0; i < 100; i++) {
    sum += x[ind];
    if (sum > 0)
        ind = ind_next_pos[ind];
    else
        ind = ind_next_neg[ind];
}

Assume the following operation latencies:

  • A memory read takes 1 cycle.
  • An addition takes 1 cycle.
  • A comparison and mux (the if selection) take 1 cycle.

For each loop, one way to compute the minimum initiation interval (II) is to follow these steps:

  1. List the operations performed in one iteration of the loop. For example, a step could be xi = x[ind] corresponding to loading the values from the array or sum += xi corresponding to the addition operation.
  2. Assign the operations to pipeline stages, assuming each operation takes one cycle. Write your answer in the form:
    Stage 0: ...
    Stage 1: ...
    Stage 2: ...    . For example, in Loop A, one possible assignment is:
    Stage 0: xi = x[ind];, Stage 1: sum += xi;, ...
  3. Identify any loop-carried dependencies.. For example, Stage 0 of iteration \(i\) depends on the value of ind from iteration \(i-1\).
  4. Compute the minimum II as the number of cycles required before the next iteration can begin. For example, if Stage 0 of iteration \(i\) depends on Stage 2 of iteration \(i-1\), the minimum II is 3 cycles.

Using this procedure, determine the minimum II for:

  1. Loop A
  2. Loop B
📐 Visual walkthrough

The minimum II is determined by how many cycles it takes to produce the values needed by the next iteration. The key loop-carried values here are sum and ind.

(a) Loop A

One valid way to break the work into one-cycle stages is:

  • Stage 0: xi = x[ind] and ind_new = ind_next[ind]
  • Stage 1: sum = sum + xi and ind = ind_new

The relevant loop-carried dependencies are:

  • The next iteration needs the updated ind in order to read x[ind].
  • The next iteration also needs the updated sum for the running accumulation.

Both of these values are available after 1 cycle of pipeline progress, so iteration i+1 can start one cycle after iteration i.

Therefore, the minimum initiation interval for Loop A is

II = 1.

(b) Loop B

For Loop B, the branch decision depends on the updated value of sum, so the dependency chain is longer. A valid stage assignment is:

  • Stage 0: xi = x[ind], ind_pos = ind_next_pos[ind], and ind_neg = ind_next_neg[ind]
  • Stage 1: sum = sum + xi
  • Stage 2: compare the new sum to 0 and select ind = (sum > 0) ? ind_pos : ind_neg

The important loop-carried dependency is the one for ind:

  • Iteration i+1 cannot begin its memory reads xi = x[ind] until iteration i has finished selecting the next ind.
  • That selection is not available until after Stage 2 of iteration i.

So Stage 0 of iteration i+1 depends on Stage 2 of iteration i, which requires a separation of 3 cycles.

Therefore, the minimum initiation interval for Loop B is

II = 3.

Final answers:

  • (a) Loop A: II = 1
  • (b) Loop B: II = 3
💡 Key Insight: II = distance between producing and consuming stages of the loop-carried dependency. Loop A: ind available after 1 cycle → II=1. Loop B: ind depends on sum → 3-stage chain → II=3.
📋 Grading Notes: - Equivalent stage names and intermediate variable names are acceptable as long as the dependency structure is correct. - Students do not need to use the exact same stage partition as the solution if their partition is valid and consistent with the stated one-cycle latencies. - For the II items, award credit based on whether the student derives the II from the loop-carried dependency, not just whether they guess the correct number. - Minor wording or formatting differences in how stages are presented should not be penalized.

Memory with unrolling
a), b), c) · 10.0pts
▼

A Vitis HLS kernel contains a loop that may be unrolled for performance:

float x[1024], y[1024];

for (int i = 0; i < 1024; i++) {
#pragma HLS UNROLL factor=UF
    y[i] = g(x[i]);
}

To support unrolling by a factor UF, the arrays must be partitioned into UF independent memory banks. Assume Vitis maps each bank into separate BRAM blocks with the following constraints:

  • Each BRAM block is 18 kbits.
  • Each array element is 32 bits (4 bytes).
  • Different banks must use different BRAM blocks, and arrays x and y must not share BRAM blocks.

How many BRAM18 blocks are required to store the arrays x and y for each of the following unroll factors:

  1. UF = 1
  2. UF = 2
  3. UF = 4
📐 Visual walkthrough

Step 1 – Size of each array

Each array has 1024 elements, each 32 bits:

1024 × 32 bits = 32768 bits = 32 kbits.

Step 2 – BRAM capacity

Each BRAM18 block stores 18 kbits. To store 32 kbits:

ceil(32 / 18) = 2 BRAM18 blocks per full array (if not partitioned).

Step 3 – Effect of partitioning into UF banks

Partitioning into UF banks divides each array into UF chunks:

Elements per bank = 1024 / UF.

Bits per bank = (1024 / UF) × 32.

BRAM18 per bank = ceil( bits per bank / 18 kbits ).

Total BRAM18 = (BRAM18 per bank) × (number of banks) × (2 arrays: x and y).

(a) UF = 1

  • Elements per bank: 1024
  • Bits per bank: 1024 × 32 = 32 kbits
  • BRAM18 per bank: ceil(32 / 18) = 2
  • Banks: 1
  • Arrays: 2 (x and y)

Total BRAM18 = 2 × 1 × 2 = 4.

(b) UF = 2

  • Elements per bank: 1024 / 2 = 512
  • Bits per bank: 512 × 32 = 16 kbits
  • BRAM18 per bank: ceil(16 / 18) = 1
  • Banks: 2
  • Arrays: 2 (x and y)

Total BRAM18 = 1 × 2 × 2 = 4.

(c) UF = 4

  • Elements per bank: 1024 / 4 = 256
  • Bits per bank: 256 × 32 = 8 kbits
  • BRAM18 per bank: ceil(8 / 18) = 1
  • Banks: 4
  • Arrays: 2 (x and y)

Total BRAM18 = 1 × 4 × 2 = 8.

Final answers:

  • (a) UF = 1 → 4 BRAM18 blocks
  • (b) UF = 2 → 4 BRAM18 blocks
  • (c) UF = 4 → 8 BRAM18 blocks
💡 Key Insight: Arrays must be partitioned into UF banks. BRAM18 = 18 Kbits = 2304 bytes. x and y are separate arrays → double the banks. For 1024 floats (4B each): UF=1 need 2 banks, UF=2 need 4, UF=4 need 8.
📋 Grading Notes: - Accept equivalent reasoning in bits, bytes, or kbits as long as the storage calculation is consistent. - Students do not need to write the ceiling expression explicitly if their final per-bank BRAM count is correct. - Arrays x and y must be counted separately; do not award full credit if the student computes the BRAM usage for only one array. - Minor arithmetic mistakes can still earn partial credit when the partitioning and BRAM-capacity reasoning are otherwise correct. - Units may be omitted if the intended BRAM18 block count is clear.

Consider the following 4-tap moving average computation:

for (i = 0; i < ny; i++) {
    yi = 0;
    for (j = 0; j < 4; j++) {
        yi += x[i+j];
    }
    y[i] = (yi >> 2);
}

Assume that x is stored in a single-port memory, so only one read from x may occur per cycle.

(a) What is the minimum possible initiation interval (II) of the outer loop?

(b) Suppose the inner loop is fully unrolled using #pragma HLS UNROLL. Does this change the minimum II? Explain why or why not.

(c) Now consider using a delay line to reduce the number of memory reads per iteration:

int xdly[4];
#pragma HLS ARRAY_PARTITION variable=xdly complete

Rewrite the loop using the delay line so that the outer loop can achieve II = 1. Your rewritten loop should read only one new value from x per iteration.

📐 Visual walkthrough

(a) Minimum II with single-port memory

Each iteration of the outer loop requires four reads: 

x[i+0], x[i+1], x[i+2], x[i+3]

With only one memory port, these four reads must occur in four separate cycles. Therefore:

Minimum II = 4

(b) Fully unrolling the inner loop

Unrolling the inner loop attempts to schedule all four additions in the same cycle. However, the memory still has only one read port, so the four reads must be serialized:

Cycle 0: read x[i+0]
Cycle 1: read x[i+1]
Cycle 2: read x[i+2]
Cycle 3: read x[i+3]

Thus, even with unrolling:

Minimum II = 4

Unrolling cannot overcome a memory bandwidth limitation.

(c) Delay-line rewrite achieving II = 1

Using a delay line, we keep the last four samples in registers and read only one new value per iteration. The delay line is fully partitioned so each element becomes an independent register: 

int xdly[4];
#pragma HLS ARRAY_PARTITION variable=xdly complete

A pipelined II=1 implementation is: 

// Initialize delay line
for (int k = 0; k < 4; k++) {
#pragma HLS UNROLL
    xdly[k] = 0;
}

for (int i = 0; i < ny; i++) {
#pragma HLS PIPELINE II=1

    // Shift delay line
    xdly[3] = xdly[2];
    xdly[2] = xdly[1];
    xdly[1] = xdly[0];

    // Read one new sample
    xdly[0] = x[i];

    // Compute 4-tap sum
    int yi = xdly[0] + xdly[1] + xdly[2] + xdly[3];

    y[i] = yi >> 2;
}

This version performs only one memory read per iteration, so the loop can be pipelined with:

II = 1

Final answers:

  • (a) II = 4
  • (b) II = 4
  • (c) II = 1 using a delay line
💡 Key Insight: Pipeline the accumulation: read → multiply → accumulate → shift. The loop-carried dependency is on the running sum.
📋 Grading Notes: - For parts (a) and (b), no explanation is required if the final answer is correct. - For part (c), accept equivalent code structures and variable names if the design clearly uses stored previous samples to reduce memory reads to one per iteration. - For part (c), an inner-loop implementation is acceptable only if it is fully unrolled. - Minor syntax differences in HLS pragmas or C indexing should not be penalized if the intended architecture and II are clear.
